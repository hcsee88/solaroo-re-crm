1. PROJECT OVERVIEW
==================================================

What This System Is:
  A full-lifecycle CRM for an engineering company that builds off-grid solar,
  hybrid solar-diesel, and battery energy storage (BESS) projects. It tracks
  the entire business cycle from initial lead to long-term maintenance.

  Lifecycle covered:
  Customer Lead -> Site Assessment -> Technical Study -> Proposal (with approvals)
  -> Contract Signing -> Project Execution (6 formal gates) -> Commissioning
  -> Asset Handover -> Operations & Maintenance (O&M)

  This is NOT a public product. It is an internal tool for ~15-30 staff.

Tech Stack Summary:
  Backend:  NestJS 10 (Node.js framework) running on Fastify, TypeScript
  Frontend: Next.js 14 (React 18), TypeScript, Tailwind CSS
  Database: PostgreSQL with Prisma ORM
  Auth:     JWT tokens stored as secure httpOnly cookies
  Queue:    BullMQ + Redis (for background jobs — currently not yet active)
  Storage:  Local disk now, AWS S3 SDK is already installed for production
  AI:       Anthropic Claude API already integrated in backend
  Email:    Resend SDK installed
  Monorepo: Turborepo (pnpm workspaces) — backend, frontend, and shared packages

4. AUTHENTICATION AND AUTHORIZATION STATUS
==================================================

--- 4A. HOW LOGIN WORKS ---

When a user logs in (POST /auth/login):
  1. Backend checks email + password (bcrypt comparison)
  2. If valid, a JWT token is generated and stored in an httpOnly cookie
  3. The browser automatically sends this cookie on every subsequent request
  4. All API requests are protected — no cookie = 401 Unauthorized
  5. GET /auth/me returns the logged-in user's profile

The JWT contains: user ID, email, name, role ID, role name.
JWT expiry: set by JWT_EXPIRES_IN env var (default: "8 hours").

--- 4B. TWO GLOBAL SECURITY LAYERS ---

Every API request passes through two global guards in this order:

  GUARD 1: JwtAuthGuard
    - Checks if a valid JWT cookie is present
    - If not: returns 401 Unauthorized immediately
    - Can be bypassed with @Public() decorator (used only for /auth/login)
    - STATUS: [DONE] — Working correctly

  GUARD 2: PermissionGuard
    - Checks if the user's role has permission for the specific action
    - Reads @RequirePermission('resource', 'action') on each route
    - STATUS: [RISK] — See critical note below

  CRITICAL RISK — AUTHZ_ENFORCE env variable:
    The PermissionGuard has a mode switch controlled by an environment variable
    named AUTHZ_ENFORCE:

    If AUTHZ_ENFORCE = "true":  Permission denied = request is BLOCKED (correct)
    If AUTHZ_ENFORCE is missing or false: Permission guard only LOGS a warning
    but DOES NOT BLOCK the request. Any logged-in user can access any endpoint.

    [RISK] If AUTHZ_ENFORCE is not set to "true" in the production server,
    every authenticated user (even a Site Supervisor) can access contracts,
    proposals, admin user lists, etc. This must be confirmed and fixed
    before any real users access the system.

--- 4C. WHAT EACH GUARD DOES AND DOES NOT PROTECT ---

JwtAuthGuard protects:     ALL routes (unauthenticated users are always blocked)
PermissionGuard protects:  Routes that have @RequirePermission decorator AND
                           AUTHZ_ENFORCE=true is set in the server environment.

[RISK] Not all controller routes have @RequirePermission decorators. Some
newer/simpler endpoints may have been written without them during rapid
development. These are open to any authenticated user regardless of role.

--- 4D. SENSITIVE FIELD PROTECTION ---

Some fields are additionally protected at the data level (not just route level):
  - marginPercent, estimatedMargin  -> only Director and Sales Manager can see
  - contractValue                   -> only Director and Finance Admin can see
  - paymentStatus                   -> only Director and Finance Admin can see

These are stripped from API responses by the AuthzService.stripSensitiveFields()
method. [RISK] This method must be explicitly called in each service. It is not
automatic. If a service returns a raw DB object without calling this method,
sensitive financial data may leak to unauthorized users.

--- 4E. OPEN REGISTRATION RISK ---

[RISK] The endpoint POST /auth/register exists. It is unclear whether it is:
  a) Restricted to SUPER_ADMIN only (safe)
  b) Unrestricted and accessible to anyone (very dangerous)

If (b), anyone with access to the network can create accounts with any role.
This must be verified and locked down immediately.

--- 4F. OTHER AUTH GAPS ---

[RISK] Default COOKIE_SECRET value is "change-this-secret" in the code.
  If this is not overridden in the production .env file, session cookies can
  be forged by anyone who knows the default. Must be set to a random string.

[RISK] No rate limiting specific to the login endpoint. The global rate limit
  (200 requests/minute) applies to all routes equally. A targeted brute-force
  attack on /auth/login is not specifically prevented.

[MISSING] No account lockout after N failed login attempts.
[MISSING] No two-factor authentication (2FA/MFA).
[MISSING] No refresh token mechanism. Users must re-login every 8 hours.
[MISSING] Login and logout events are not written to an audit log.

5. ROLE AND PERMISSION STRUCTURE
==================================================

--- 5A. THE 11 ROLES ---

Each user has exactly one role. The role determines what they can see and do.

  Role Name               | Who Uses It
  ------------------------|----------------------------------------------
  SUPER_ADMIN             | System owner — manages users, roles, and settings
  DIRECTOR                | Top management — sees everything including margins
  SALES_MANAGER           | Runs sales team — sees full pipeline, approves proposals
  SALES_ENGINEER          | Individual sales — manages own deals and proposals
  PROJECT_MANAGER         | Leads project delivery — manages gate progress
  DESIGN_ENGINEER         | Does technical design — works on assigned projects
  PROCUREMENT             | Buys equipment — manages vendors, POs, deliveries
  SITE_SUPERVISOR         | On-site execution — updates site logs and checklists
  COMMISSIONING_ENGINEER  | Commissions systems — testing, handover, asset register
  OM_ENGINEER             | Operates and maintains — tickets, warranty, maintenance
  FINANCE_ADMIN           | Handles money — contracts, invoices, payment status

--- 5B. SCOPE LEVELS (WHAT EACH ROLE CAN SEE) ---

The system has 4 visibility scopes:

  "own"      = Can only see records they created or own
  "team"     = Can see records owned by anyone in their same role
  "assigned" = Can see records they have been explicitly added to
  "all"      = Can see everything regardless of ownership

Examples:
  Sales Engineer: opportunity:view:own   (only sees their own deals)
  Sales Manager:  opportunity:view:all   (sees all deals in the pipeline)
  Director:       contract:view:all      (sees all contracts and values)
  Project Manager: project:view:assigned (sees projects they are assigned to)

--- 6C. DATABASE CONCERNS ---

[RISK] Proposal.createdByUserId is stored as a plain string with no foreign key
  constraint to the User table. If a user is deleted, the reference becomes an
  orphan with no error. This can cause reporting or display issues.

[RISK] File paths (fileUrl) stored in the database point to local disk paths.
  When the system migrates to AWS S3 (which is the plan), all existing records
  will have broken file paths. A migration strategy is needed.

[RISK] OpportunityMember and Team models exist in the schema but there is no
  API endpoint or UI to add users to opportunities or teams. This means the
  "assigned" scope in the permission system cannot work for non-project resources
  (e.g., a Sales Engineer cannot be added to a specific opportunity to gain
  shared access with a colleague).

[MISSING] No consistent soft-delete strategy. Some models have isActive, others
  do not. Hard-deleted records cannot be recovered.

[MISSING] No general field-level audit log. No table records "who changed what
  field, from what value, to what value, at what time." Only stage changes
  (OpportunityStageHistory) and approval decisions (ProposalApproval) are tracked.

9. MAJOR GAPS AND RISKS
==================================================

CRITICAL — Fix Before Any Production Use:

  [RISK-1] AUTHZ_ENFORCE not confirmed as "true" in production.
    Without this, all authenticated users have full access to all endpoints.
    Action: Set AUTHZ_ENFORCE=true in production environment. Verify immediately.

  [RISK-2] POST /auth/register may be unrestricted.
    If open, anyone can register accounts and assign themselves any role.
    Action: Add SUPER_ADMIN permission check or disable the endpoint entirely.

  [RISK-3] Default COOKIE_SECRET is "change-this-secret" in source code.
    If the .env file does not override this, session cookies can be forged.
    Action: Set a cryptographically random COOKIE_SECRET in every environment.

  [RISK-4] Test user accounts exist in production DB after seeding.
    seeded test users (see@pekatgroup.com, director@pekatgroup.com, etc.)
    with known passwords should be deactivated before going live.
    Action: Run a script to deactivate all seeded test users in production.

HIGH — Fix Soon:

  [RISK-5] Sensitive field stripping (margin, contract value) is not automatic.
    Each service that returns financial data must explicitly call
    stripSensitiveFields(). If any service skips this call, margin data
    or contract values can be seen by users who should not have access.
    Action: Audit every service that returns proposal or contract data.

  [RISK-6] File download path traversal risk.
    The documents download endpoint streams a file using a path stored in the DB.
    If this path is used directly without validation, a crafted DB record could
    allow reading arbitrary server files.
    Action: Validate and sanitize fileUrl before constructing file path.

  [RISK-7] Proposal approver matching is fragile.
    The system finds the Sales Manager and Director by matching their role's
    display name text (e.g., "contains 'Sales Manager'"). If that role is
    renamed in the admin panel, the approval chain silently breaks.
    Action: Match by role.name (exact code enum) instead of display name text.

  [RISK-8] "Assigned" scope cannot work for opportunities.
    The permission system has "assigned" scope for opportunities (meaning a
    user can see an opportunity they are explicitly added to). But there is no
    API endpoint or UI to add users to opportunities. OpportunityMember records
    can never be created. This makes the scope non-functional.
    Action: Build an "Add Team Member" API and UI for opportunities.

  [RISK-9] No email notifications for approvals.
    When a proposal is submitted for approval, the designated approver receives
    no email or notification. They must manually check the system.
    The background worker (BullMQ) is not yet functional.
    Action: Implement the worker job that sends approval request emails.

MEDIUM — Address in Next Development Phase:

  [GAP-10] No test coverage at all.
    Zero test files exist in the entire codebase. Any code change may break
    existing features with no automated safety net.
    Action: Write unit tests for proposal approval logic, stage transitions,
    and auth endpoints at minimum.

  [GAP-11] No CI/CD pipeline.
    There is no automated process to build, test, and deploy the application.

File: apps/api/src/common/authz/permission.guard.ts
Why: Shows the critical AUTHZ_ENFORCE mode — audit-only vs enforcing.

  if (!this.configService.get('AUTHZ_ENFORCE')) {
    // AUDIT MODE: logs but does NOT block the request
    this.logger.warn(`AUTHZ_AUDIT: user=${user.id} action=${resource}:${action}`);
    return true; // <-- request goes through regardless of permission
  }
  // ENFORCE MODE: blocks if permission is denied
  return this.authzService.hasPermission(user, resource, action);

  private async findApprovalUsers() {
    const [salesManagers, directors] = await Promise.all([
      this.prisma.user.findMany({
        where: {

          role: { displayName: { contains: 'Sales Manager', mode: 'insensitive' } }
        }
      }),
      this.prisma.user.findMany({
        where: {
          isActive: true,
          role: { displayName: { contains: 'Director', mode: 'insensitive' } }
        }

    // If role is renamed or display name changes — breaks silently.
    return {
      salesManagerId: salesManagers[0]?.id ?? null,
      directorId:     directors[0]?.id ?? null,
    };
  }

  export async function uploadFile(path, file, fields) {
    return new Promise((resolve, reject) => {
      const reader = new FileReader();
      reader.onload = async () => {
        const base64 = (reader.result as string).split(',')[1]; // strip data: prefix
        const response = await apiClient.post(path, {
          ...fields,
          fileBase64:    base64,
          fileName:      file.name,
          mimeType:      file.type,
          fileSizeBytes: file.size,
        });
        // ...
      };
      reader.readAsDataURL(file);
    });
  }

  // No virus scanning. MIME type taken from browser (easily spoofed).
