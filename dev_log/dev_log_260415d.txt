B. PROJECT_ENGINEER ROLE SECURITY AUDIT
============================================================

Requested audit: project edit scope, proposal visibility, scope enforcement.
Full analysis performed against: projects.service.ts, projects.dto.ts,
proposals.service.ts, documents.service.ts, permissions.seed.ts,
authz.service.ts, sensitive-fields.ts, projects.controller.ts.

FINDINGS SUMMARY

  R1 — CRITICAL  project:edit:assigned gives PE write access to projectManagerId
                 PE can reassign PM or self-promote

  R2 — CRITICAL  budgetBaseline and budgetUpdated in UpdateProjectSchema
                 PE has zero budget read access but can WRITE these fields
                 (no field-level guard in update() method)

  R3 — HIGH      Gate status endpoint uses project_gate:edit guard
                 UpdateGateStatusSchema allows status: 'APPROVED'
                 Service has no approval-authority check → PE can approve gates
                 and have their ID stamped as approver

  R4 — HIGH      Deliverable update allows status: 'APPROVED'
                 No role check before stamping approvedById → PE can self-approve

  R5 — HIGH      project status field (ACTIVE/ON_HOLD/COMPLETED/CANCELLED)
                 writable by PE → can cancel or complete a project

  R6 — MEDIUM    commercialSummary returned to PE with no gate
                 Free-text field routinely contains CAPEX/margin prose

  R7 — MEDIUM    Proposal + opportunity scope for PE resolves via OpportunityMember
                 PE is assigned via ProjectMember (not OpportunityMember)
                 If admin adds PE only as ProjectMember → they see zero proposals
                 and zero opportunities in list views (correct design intent,
                 but the traversal path is not through ProjectMember → this is
                 a gap if admin expects project membership to imply proposal access)

  R8 — MEDIUM    pmoStatusNote, lastPmoReviewAt writable by PE (PMO governance)

  R9 — MEDIUM    designLeadId, procurementLeadId writable by PE (staffing assignments)

  R10 — MEDIUM   estimatedSavings, paybackYears returned to PE — not in stripping
                 policy, but economically revealing — BUSINESS DECISION NEEDED

CONFIRMED CORRECT
  - estimatedCapex, estimatedMargin, marginPercent → correctly null for PE
    (gated on margin:view_estimated permission in proposals.service.ts)
  - Project list + detail scope: correctly checks ProjectMember.userId
  - Document scope: correctly traverses project.members path → PE sees project docs
  - Gate approval permission exists (project_gate:approve) but is not called
    inside updateGateStatus() — permission exists, enforcement missing

RECOMMENDED FIXES (not yet implemented — pending next session)
  1. projects.service.ts update(): add role-based field guard before Prisma call
     PE-safe fields: name, notes, ragStatus, currentBlocker, blockerOwnerId, blockerDueDate
     Restricted (throw ForbiddenException if PE sends them):
       projectManagerId, designLeadId, procurementLeadId,
       budgetBaseline, budgetUpdated, status, pmoStatusNote, lastPmoReviewAt

  2. updateGateStatus(): check project_gate:approve before allowing APPROVED/REJECTED
  3. updateDeliverable(): check project_gate:approve before allowing APPROVED
  4. proposals.service.ts findById(): gate commercialSummary on canViewMargin
  5. Proposal findAll() scope: add project.members traversal path for PE
  6. Decide on estimatedSavings / paybackYears stripping (business decision)
