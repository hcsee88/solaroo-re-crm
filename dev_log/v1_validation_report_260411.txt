RISKS / POSSIBLE BREAKAGES
===========================

RISK 1 — Design Lead has no permissions seeded [HIGH]
  The permissions.seed.ts maps permissions to role names.
  DESIGN_LEAD was not added. When AUTHZ_ENFORCE=true, a Design Lead user
  will be denied access to all endpoints.
  Fix: Add DESIGN_LEAD permission grants to permissions.seed.ts, re-run
       pnpm db:seed.

RISK 2 — blockerOwnerId has no UI [LOW]
  The field exists in the DB and DTO but the RagPanel only lets users set
  ragStatus, currentBlocker, and blockerDueDate. The blockerOwner user
  must be set directly in the DB or via a future UI picker.
  Impact: blockerOwner will always be null from the UI. Non-breaking.

RISK 3 — Stale role data mid-session [LOW]
  useCurrentUser has staleTime: 5 * 60 * 1000. If an admin changes a user's
  role while they are logged in, the sidebar/guards won't reflect this for
  up to 5 minutes. Acceptable for V1.

RISK 4 — ragStatus groupBy _count access [MEDIUM]
  The reporting service uses row._count.id in groupBy results. If Prisma
  returns _count differently in this context, all RAG counts would be 0.
  Observed in live data: GREEN=1, AMBER=0, RED=0 — appears correct but only
  verified with one project.

RISK 5 — GateDeliverable field name assumptions [MEDIUM]
  Reporting query uses isRequired, status, gate.status, gate.project.status.
  If any field name differs in the actual schema, the count silently returns 0.
  Should be verified against schema/projects.prisma GateDeliverable model.
