SECTION 1 — WHAT THIS SYSTEM IS
---------------------------------
Solaroo RE CRM is an internal, single-company platform built for Pekat Teknologi
Sdn Bhd. It is NOT a SaaS product. It covers the full lifecycle of off-grid solar,
hybrid solar-diesel, and BESS (Battery Energy Storage System) projects — from
first customer contact through O&M handover.

It also functions as a light ERP (procurement, invoicing milestones, contract
tracking) and a document control system.

Typical projects handled: plantation solar, island microgrids, resort off-grid,
C&I solar + diesel hybrid, weak-grid BESS. Project values range from RM 500K to
RM 10M+.

Team size: 15 users now. Expected ~30 in 2 years.

Business lifecycle covered:
  Account → Site → Opportunity → Proposal → Contract → Project
  → Commissioning → Asset Register → O&M

SECTION 7 — SECURITY AND PERMISSIONS
--------------------------------------
Two-layer security model:

Layer 1 — Backend (hard enforcement):
  - JwtAuthGuard: validates httpOnly cookie on every request
  - PermissionGuard: checks user.role.permissions against @RequirePermission()
  - AUTHZ_ENFORCE=true: returns HTTP 403 on any permission violation
  - All mutations validated with ZodValidationPipe at controller level

Layer 2 — Frontend (UI gating):
  - Single config file: apps/web/src/lib/role-ui.ts
  - Pure functions: canSeeProposalCapex(), canSeeOpportunityValue(), etc.
  - Sidebar nav items filtered per role (getNavItemsForRole)
  - Table columns hidden per role (CAPEX, Margin, pipeline values)
  - Dashboard sections conditionally rendered per role

SECTION 9 — KEY BUSINESS RULES IMPLEMENTED
--------------------------------------------
1. Opportunity stage transitions: controlled state machine (no arbitrary jumps)
   Defined in @solaroo/workflows package. Transitions validated server-side.

2. Proposal versioning: proposals are immutable after approval. Changes require
   a new version. Every version references a frozen ProposalAssumptionSet.

3. Won opportunity → project: a project record is auto-created when opportunity
   stage changes to WON. Never created manually.

4. Project code format: SRE{YY}-{SITE_PREFIX}{NNN} (e.g. SRE26-PSO002)

5. Auto-generated codes: ACC-XXXX (accounts), SITE-XXXX (sites),
   OPP-YY-NNN (opportunities), VEN-XXXX (vendors), PRD-XXXX (products),
   PO{YY}-XXXX (purchase orders)

6. Audit logging: all status changes, approvals, and key edits write to AuditLog.

7. Assets: only created at commissioning/handover — never before.

8. Warranty: captured per installed asset at serial number level.

9. Gate tracker: structured checklists with approval states per gate.
   Gate closure is auditable.

10. RAG status: projects have GREEN/AMBER/RED status with blocker tracking
    (blocker description, owner, due date) managed via PMO module.

SECTION 13 — ARCHITECTURE DECISIONS AND RATIONALE
---------------------------------------------------
Decision: Modular monolith, not microservices
  Rationale: 15-30 users. All domains share one DB. No distributed transaction
  complexity needed. One deployment, one codebase, one Prisma schema.

Decision: Prisma split schema files
  Rationale: Single schema.prisma becomes unreadable at 56 models. Domain
  separation (auth.prisma, crm.prisma, etc.) makes maintenance tractable.

Decision: httpOnly cookie for JWT (not localStorage)
  Rationale: XSS protection. Frontend never touches the token directly.

Decision: Two-layer security (backend AUTHZ + frontend role-ui.ts)
  Rationale: Backend AUTHZ blocks unauthorized API calls (hard enforcement).
  Frontend role-ui.ts hides irrelevant UI (UX + accidental exposure prevention).
  Both layers independent — bypassing frontend doesn't bypass backend.

Decision: Permission to access endpoint ≠ permission to see every field
  Rationale: Commercial sensitivity. A Project Manager can access /proposals
  for project context but should not see margin % or CAPEX. Field-level
  visibility controlled separately from endpoint access.

Decision: Immutable proposal versions
  Rationale: Solar project proposals involve months of negotiation. Version
  history must be auditable. Approved versions can never be modified.

Decision: Frozen assumption sets on proposals
  Rationale: Proposal pricing depends on assumptions (grid tariff, diesel price,
  solar irradiation, equipment specs). If assumptions change, a new version is
  required so old vs new can be compared.

Decision: Won opportunity → project auto-creation
  Rationale: Prevents data inconsistency. A won opportunity always has exactly
  one project. Manual creation would allow won opportunities without projects.
