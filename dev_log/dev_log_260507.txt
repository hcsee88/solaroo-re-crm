4. OPPORTUNITY EDIT BUG FIX — every save was failing with HTTP 400
   - Root cause: nextAction / nextActionDueDate / lastStatusNote were
     declared .optional() but not .nullable() in UpdateOpportunitySchema.
     Frontend edit form sends `null` when fields are blank, which Zod
     rejected → BadRequestException → 400.
   - Secondary bug: in opportunities.service.ts, the date ternary
     `dto.nextActionDueDate ? new Date(...) : undefined` mapped null →
     undefined (no-op), so even after the schema was fixed, clearing a date
     wouldn't actually clear the column.
   - Fix: added .nullable() to the 5 V1 tracking fields; rewrote date
     mappers as 3-way (null → null, undefined → undefined, string → parse).
   - Verified end-to-end: PATCH with nulls returns 200 and persists null;
     PATCH that sets values, then re-PATCH that clears, both work.

5. RAILWAY AUTO-DEPLOY RECONNECTED + DOCKERFILE HARDENED
   - Discovered after a push that Railway hadn't auto-deployed since
     2026-04-25. Root cause: Auto deploy was DISABLED on both API and Web
     services (visible in Railway → Settings → Source). User re-enabled.
   - Triggered redeploy via empty commit (commit 23376bd) — both services
     picked it up.
   - Then opportunities list went 500: error logs showed
     `column opportunities.designEngineerId does not exist`. Root cause:
     production startup used `prisma migrate deploy` and no migration file
     existed for our new column. Schema diverged → API queries failed.
   - FIX (commit fee0851):
     Replaced Dockerfile.api startup CMD:
       OLD: prisma migrate deploy ... || echo skipped
       NEW: prisma db push --skip-generate (no --accept-data-loss)
     Behaviour:
       - Adds new columns/tables/indexes silently       ✓
       - REFUSES & exits non-zero on destructive change ✓
         (column drop, type narrowing, NOT-NULL without default…)
     Combined with the seed script (idempotent upsert only on roles/perms/
     test users — never touches business data), business data CANNOT be
     wiped by a normal deploy. To do a destructive change, an operator
     must run db push --accept-data-loss manually outside the container.
   - Deploy fee0851 logs confirm:
       "🚀 Your database is now in sync with your Prisma schema. Done in 286ms"
       "✓ 14 roles seeded ... ✓ 13 test users seeded"
     → Missing designEngineerId (and other additive sales-pipeline cols)
     were applied automatically on container start. Existing rows untouched.
     User's opportunities became visible again.

6. AUDIT v2 — auto-capture + "Created by / Last edited by" pill (commit 8c3eb70)
   - User asked which option was better:
       A. Add createdBy/updatedBy columns to every business table
       B. Build a global AuditInterceptor + read from existing AuditLog
   - Recommended B (no schema risk, lower engineering cost, future-proof).
   - Implementation (~90 min):
     • apps/api/src/common/audit/audit.interceptor.ts (new, ~120 lines)
       - NestJS APP_INTERCEPTOR registered globally via AuditModule
       - Auto-records every successful POST /api/{resource}  → 'created'
                                       PATCH /api/{resource}/:id → 'updated'
       - Top-level routes only — sub-routes (transitions, approvals) deliberately
         skipped because services already audit those with rich metadata
       - Routes covered: opportunity, account, site, contact, project,
         contract, proposal, proposal_version, activity
       - Captures: actor.id, resource, resourceId, action, metadata.fields
         (list of submitted field names — not values, to keep audit log lean)
     • apps/api/src/common/audit/audit.service.ts
       - getCreatedBy(resource, resourceId) — first audit_logs row with action='created'
       - getLastEdit(resource, resourceId) — most recent audit_logs row
       - getEditMeta(resource, resourceId) — combined fetch in single round trip
     • apps/api/src/modules/admin/admin.controller.ts
       - GET /api/admin/audit/edit-meta?resource=&resourceId= → no permission
         gate (the resource itself is already gated by its own module's perms)
     • apps/web/src/components/audit/edit-meta-pill.tsx (new, ~85 lines)
       - Renders "Created by X · 23 Apr 2026 · Last edited by Y · 18 minutes ago"
       - Silent on error — never blocks the page
     • Wired into 4 detail page headers: opportunities, projects, contracts,
       proposals
   - Honest caveats noted in commit:
       - Existing pre-interceptor records have no creator entry (history
         starts now)
       - No oldValue/newValue diff captured for plain edits yet (only field
         names list); state transitions still capture rich diff via existing
         service-layer audit calls
       - Sub-routes deliberately skipped to avoid double-audit
   - Storage estimate: <1k rows/day at current team size (15→30 users) —
     trivial for Postgres.
