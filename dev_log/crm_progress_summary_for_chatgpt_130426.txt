1. EXECUTIVE SUMMARY
============================================================

WHAT THE CRM CURRENTLY IS:
  Solaroo RE CRM is an internal single-company platform for managing the full
  lifecycle of off-grid solar, hybrid solar-diesel, and BESS projects. It covers
  lead/account management through to technical assessment, proposal generation,
  contract, project execution (6-gate governance), commissioning, and O&M.
  It also functions as a light ERP and document control platform.

  Stack: NestJS (Fastify) API + Next.js 14 frontend + PostgreSQL via Prisma.
  Monorepo with Turbo (apps/api, apps/web, packages/db, packages/types).
  Auth: JWT httpOnly cookie. RBAC: 13 roles, 330+ permission entries, 4 scopes.

CURRENT DEVELOPMENT STAGE:
  Early V1 — core CRM backbone is operational. The sales pipeline (accounts →
  opportunities → proposals), project execution (projects, gates, milestones,
  issues, risks), document management, dashboard, and auth/admin are all working.
  Several modules exist as stubs (procurement, contracts service, O&M, AI,
  notifications). Frontend pages exist for all major modules but many are
  read-only or partially implemented.

WHAT HAS IMPROVED RECENTLY (last 2 sessions):
  - Self-service password change (logged-in users via topbar)
  - DESIGN_LEAD role fully implemented with approval permissions
  - PMO integration: PMO_MANAGER role, PMO schema fields on projects/gates/
    milestones, GET /reporting/pmo endpoint, /pmo portfolio page in the UI
  - AUTHZ_ENFORCE=true enabled — permission enforcement is now live
  - Access Denied page (/forbidden) added — 403 now shows proper UI
  - Reporting:view permission fixed for operational roles (they were getting
    403 on the dashboard after enforcement was turned on)
  - start-dev.bat updated to survive Docker-not-running gracefully

V1 DIRECTION CLARITY:
  V1 direction is becoming clear. The system is focused on:
  - Sales pipeline: accounts → opportunities → proposals (with versioning)
  - Project lifecycle: 6-gate governance, RAG status, blocker tracking
  - Cross-portfolio visibility: dashboard + PMO portfolio page
  - Role-based access with enforced permissions
  V1 is deliberately not trying to finish procurement, O&M, or AI.

BIGGEST CURRENT STRENGTHS:
  1. Data model is comprehensive and domain-accurate for solar/BESS projects
  2. Permission system is sophisticated (scoped, field-level, cacheable, enforced)
  3. Opportunity stage flow and proposal versioning are implemented correctly
  4. 6-gate project governance is fully modelled and partially implemented
  5. PMO has been sensibly integrated as a thin governance layer, not a system
  6. No public registration — admin-only user creation is correct for internal tool

BIGGEST CURRENT RISKS:
  1. Several critical modules are stubs (procurement, contracts, O&M, AI) —
     these are on the critical path for later project lifecycle stages
  2. Scope enforcement (own/assigned/team) is declared but NOT fully validated
     at service layer — INFERENCE: services may return too much data for
     non-all-scope roles (UNCONFIRMED per-service)
  3. Field-level sensitive data stripping (margin, cost, payment_status) is
     implemented in AuthzService but it is UNCONFIRMED whether all service
     responses actually call stripSensitiveFields()
  4. AUTHZ_ENFORCE is now true but full role-by-role testing has only been
     done partially (blocked roles now redirect to /forbidden, but scope
     overreach in data responses has not been validated)

============================================================

IMPLEMENTED IN CURRENT FORM:

  the CRM, not as a standalone module.

PMO ELEMENTS THAT EXIST:

  Schema level:
    Project.pmoStatusNote      (String?) — PMO's governance note
    Project.lastPmoReviewAt    (DateTime?) — last formal review
    ProjectGate.pmoFlagged     (Boolean, default false)
    ProjectGate.pmoComment     (String?)
    ProjectGate.pmoReviewAt    (DateTime?)
    Milestone.baselineDate     (DateTime?) — frozen original date for drift detection

  Role level:
    PMO_MANAGER role (dedicated, separate from Director and PM)
    23 permission entries — cross-portfolio view, gate approval, milestone
    and issue create/edit. No contract/invoice/margin access.

  API level:
    GET /reporting/pmo — 10 parallel queries producing:
      quickCounts (8 metrics), ragBreakdown, gateDistribution,
      projectsWithBlocker, pendingGateApprovals, overdueDeliverables,
      overdueMilestones, stalledProjects (14-day threshold), criticalOpenIssues

  Frontend level:
    /pmo page with 8 quick-count cards, RAG health bar, gate distribution
    bars, and 7 detail tables. Sidebar nav item (ShieldCheck icon).

WHAT IS MISSING FOR PRACTICAL PMO USE:
  - PMO gate approval action from the /pmo UI (currently read-only;
    approval done from project detail page)
  - Structured review log (pmoStatusNote overwrites; no history per review)
  - PMO-specific notifications (gate submitted, blocker escalation)
  - Baseline vs actual schedule comparison chart (data model supports it,
    chart not built)

EMBEDDED VS SEPARATE:
  Embedded. PMO data lives on existing Project, ProjectGate, and Milestone
  models. PMO_MANAGER is a role with broad view permissions. The /pmo page
  is a portfolio aggregation view in the reporting section of the CRM.
  There is no separate PMO module or separate PMO database.

BEST-FIT APPROACH BASED ON REPO:
  Current approach is correct for a 15-person company. PMO should remain
  embedded and governance-oriented — not a project manager replacement.
