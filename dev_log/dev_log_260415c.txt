Dev Log — 2026-04-15 (c)
Session: Global Search Implementation
========================================

FEATURE: Global Search (⌘K Command Palette)
---------------------------------------------

PROBLEM
The TopBar search pill was a styled, non-functional <button> with no onClick handler,
no state, and no API call. Clicking or pressing ⌘K did nothing.

SOLUTION
Implemented a full global search with:
  - Backend: NestJS SearchModule (GET /search?q=...)
  - Frontend: SearchModal command-palette component
  - TopBar wired up with onClick and ⌘K/Ctrl+K global shortcut

========================================
BACKEND — apps/api/src/modules/search/
========================================

NEW FILES
  search.service.ts    — Queries 6 resources, scope-aware, 5 results per group
  search.controller.ts — GET /search?q= with ZodValidationPipe, requires auth
  search.module.ts     — Module registration

MODIFIED
  app.module.ts        — SearchModule added to imports array

SEARCH COVERAGE
  Resource      Searched fields                     Scope enforcement
  ------------- ----------------------------------- ----------------------------
  Account       name, accountCode, industry, region  any (no scope filter; all roles with account:view get all)
  Contact       firstName, lastName, email,          any
                jobTitle, phone
  Site          name, siteCode, address, region      any
  Opportunity   title, opportunityCode, summary,     own / assigned / team / all
                account.name
  Proposal      title, proposalCode,                 own / assigned / team / all
                opportunity.account.name
  Project       name, projectCode, account.name      assigned / all

SCOPE FILTERS (match exactly what service layer uses)
  Opportunity: own → ownerUserId=me; assigned → ownerUserId OR members.userId; team → owner.role.name
  Proposal:    own → createdByUserId=me; assigned → opportunity.(ownerUserId OR members.userId)
  Project:     assigned → projectManagerId OR members.userId

RESPONSE SHAPE
  {
    query: string,
    total: number,
    groups: [
      { type, label, items: [{ id, type, code, title, subtitle, url }] }
    ]
  }
  Groups ordered: opportunities → projects → proposals → accounts → contacts → sites
  Empty groups are omitted. 5 results max per group.

SCHEMA FIXES (verified against Prisma schema before writing)
  - Contact uses firstName/lastName (not name); accounts via AccountContact[]
  - Opportunity owner field is ownerUserId (not ownerId)
  - Proposal uses createdByUserId; account goes through opportunity.account
  - OpportunityMember.userId, ProjectMember.userId confirmed in auth.prisma
  - Site.address, Site.account direct relation confirmed

FRONTEND — apps/web/src/components/layout/
========================================

NEW FILE
  search-modal.tsx  (~210 lines)

FEATURES
  - Full-screen backdrop with modal panel (max-w-xl, top: 15vh)
  - Auto-focus input on open; clears state on each open
  - 280ms debounce before API call (avoids hammering API while typing)
  - Loading spinner replaces search icon while fetching
  - Grouped results: label + count badge per group
  - Each result: type icon bubble (color-coded) + title + code chip + subtitle
  - Keyboard navigation: ↑↓ moves activeIndex, ↵ opens record, Esc closes
  - Mouse hover also sets activeIndex (synced with keyboard)
  - Active item: blue tinted background + colored left border + ↵ hint
  - Empty state: icon + "No results for X" + tip text
  - Prompt state: "Type at least 2 characters" when input is empty
  - Error state: red message if API fails
  - Footer bar: result count + keyboard hint legend (↑↓ / ↵ / Esc)
