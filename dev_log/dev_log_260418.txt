Dev Log — 2026-04-18
Session: Contracts + Award→Handover module + 10 follow-ups (3 rounds)
=============================================================

OVERVIEW
--------
This was one long session that built the Contracts module from a stub into
a full Award→Handover workflow, then layered 10 follow-ups across 3 rounds
of "do all of the above". Total outcome: a working commercial-execution
spine with audit trails, variance reporting, document linking, saved
views, PMO widgets, and a daily notification digest.

This log is honest. It documents the ugly bits — wrong assumptions, two
schema corrections needed, a subtle build-config conflict, and the
deliberate scope cuts. ChatGPT (or another reviewer) should be able to
spot anything that looks suspect after reading this.

HONEST PROBLEMS THIS SESSION CAUSED + HOW THEY WERE FIXED
=============================================================

1. SCHEMA: Contract had no Opportunity relation at all
   The original schema had `opportunityId String?` but no `opportunity
   Opportunity?` relation field. I noticed this only after the API build
   failed with `'opportunity' does not exist in type 'ContractSelect'`.
   Added the relation to contracts.prisma + the back-ref on Opportunity
   in crm.prisma. Cost: ~5 min, one rebuild.

2. SCHEMA: Inverse relations needed adding for new Contract FKs
   Added siteId, proposalVersionId, projectManagerCandidateId,
   handoverCompletedById to Contract. Each needs an inverse on the
   referenced model:
     - User: contractsAsPmCandidate, contractsHandedOver
     - Site: contracts
     - ProposalVersion: contracts
   Easy to forget. Caught at first db push. No data loss because all
   additive (nullable + new enum value).

3. TYPESCRIPT: scope context required undefined, not null
   First contracts.service build had 14 type errors all of the same
   shape: `before.foo?.id ?? null` when the AuthzService scope context
   types `string | undefined`. Single replace_all fixed all 14. This is
   a code-style decision in AuthzService — `null` would work fine if the
   types allowed it. Worth aligning later.

4. NESTJS: nest-cli.json entryFile getting overwritten
   `nest build --webpack` (which we use for production via Dockerfile.api)
   silently changed `entryFile` from `apps/api/src/main` to just `main` in
   nest-cli.json. The next `nest start --watch` (dev, no webpack) couldn't
   find the entry because tsc-watch put it at `dist/apps/api/src/main.js`.
   Fix: pinned both `entry` and `output.filename` inside webpack.config.js
   so webpack ignores nest-cli.json, and restored `entryFile` to its
   tsc-friendly value. Documented in webpack.config.js.

5. WINDOWS: Prisma generate fails when dev is running
   `prisma generate` rewrites query_engine-windows.dll.node, and the
   running NestJS dev process holds an OS lock on it. Result: every
   schema push during dev throws EPERM. Fix: kill node procs first,
   regen, restart dev. Not great DX but it's a Windows file-locking
   reality, not a code bug.

6. TURBO: pnpm db:seed swallowed the real error
   First seed attempt failed with a generic "Validation Error Count: 1"
   line. Turbo had truncated the actual Prisma error
   ("Environment variable not found: DATABASE_URL"). Had to run the seed
   directly inside packages/db to see it. Root cause: tsx@4 doesn't
   auto-load .env. Fix: added --env-file=.env to the db:seed script in
   packages/db/package.json. Now works through turbo.

7. TAILWIND JIT: dynamic class names get stripped
   In the audit admin page's <Field> helper I had used
   `className={\`md:col-span-${colSpan}\`}`. JIT can't see those at build
   time, so the classes wouldn't ship. Switched to a static lookup map.
   Worth grep'ing the rest of the codebase for the same pattern at some
   point.

8. exactOptionalPropertyTypes strictness
   Hit twice this session. tsconfig has `exactOptionalPropertyTypes: true`,
   which means `error?: string` is NOT assignable from `error: string |
   undefined`. Both errors looked confusing at first glance. The fix is
   always to widen the prop type to `error?: string | undefined`.
   Mechanical, but a reviewer should note that it's a strictness toggle
   we've opted into and live with.

9. SUPER_ADMIN cannot see contracts (correct, but surprising)
   The smoke tests after each round returned 403 for the see@pekatgroup.com
   user even though everything looked wired up. The seed convention is
   that SUPER_ADMIN only has user_admin + role_admin grants — they're
   meant for system administration, not business data. Test users
   (DIRECTOR, PMO_MANAGER, etc., password Test@1234) are the right
   accounts to verify the new UIs. Documented this in the round-1 summary.

10. Worker uses setInterval, NOT BullMQ
    The original suggestion was a BullMQ scheduled job. I deliberately
    chose setInterval instead because:
      - The Hobby Railway plan doesn't have Redis available out of the box
        (Redis service was created but never deployed in this project).
      - Local dev has no Redis either.
      - Adding Redis would block setup for any new contributor.
    Tradeoff: setInterval doesn't survive worker restart cleanly (state
    lives in lastDigestSentAt, so a restart just delays the next digest
    by up to 30 min, no duplicates). When the team adopts Redis, swap the
    loop body into a BullMQ job — the digest function itself is reusable
    as-is. This is called out in the file header comment.

DELIBERATE V1 SCOPE CUTS — WHAT WE DIDN'T DO
=============================================================

- Contract modules: no edit history of the contract value or terms (the
  AuditService catches the BIG transitions but field-level diffs on
  arbitrary edits aren't recorded). Add when needed.
- Milestones: no rich UI for editing existing rows (just add + toggle
  done + delete-if-not-achieved). Edit-in-place would be V2.
- Invoices: no PDF generation. We track the invoice number / amount /
  status but the actual document is uploaded via the Documents panel
  (Document.contractId), so the linkage is there.
- LinkedDocsSection extended only to accept "contract" contextType. The
  approve/reject buttons show for all contexts but only PENDING revisions.
  We do NOT pre-filter by role — server returns 403 if the user lacks
  document:approve and the UI surfaces an error toast. Conservative.
- SavedView: only wired into /contracts list. /opportunities, /projects,
  /audit etc. all support the same SavedViewsBar component but aren't
  wired up yet — same 5-line addition each.
- Notification digest: in-app only, no email. The worker creates a
  notification of type "digest" that summarises the count + breakdown by
  type and links back to /notifications. Email would require an SMTP /
  Resend wiring (the env shows Resend is in apps/api/package.json but no
  service uses it yet).
- AuditService never records read operations. Only state transitions and
  member changes. Field-level diffs happen only when explicitly written
  with field/oldValue/newValue (gate status, contract status, blocker
  ownerId, document approval).
- contract:approve permission is granted to DIRECTOR + SALES_MANAGER for
  the AWARDED transition. invoice_milestone:create lets FINANCE_ADMIN +
  DIRECTOR add invoices but does NOT let project members do it. PMs can
  view via assigned scope but not create. Worth reviewing with the
  business if PMs should be able to draft invoices.
