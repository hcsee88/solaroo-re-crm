PART 1 — POST-PMO HARDENING
----------------------------

AUTHZ_ENFORCE=true
- Set AUTHZ_ENFORCE=true in apps/api/.env
- Restarted API. PermissionGuard now hard-blocks (returns 403) instead of audit-only.

Issue: 6 operational roles got "Access Denied" on login
- Affected: salesengineer, designengineer, procurement, sitesupervisor, commissioneng, omengineer
- Root cause: dashboard calls GET /reporting/dashboard — these roles were missing reporting:view permission
- Fix: Added reporting:view:assigned to all 6 roles in packages/db/src/seeds/permissions.seed.ts
- Re-ran pnpm db:seed to apply

Issue: 403 responses showed a blank page
- Root cause: api-client.ts only handled 401 (redirect to /login). 403 was an unhandled rejection with no UI.
- Fix: Added 403 handler in api-client interceptor → window.location.href = "/forbidden"
- Created apps/web/src/app/forbidden/page.tsx — static server component with lock icon, explanation, nav buttons
- Updated apps/web/src/middleware.ts to add "/forbidden" to PUBLIC_PATHS (no auth required to view it)

PART 2 — ROLE-BASED UI VISIBILITY (V1)
---------------------------------------

Design principle: permission to access an endpoint ≠ permission to show every field or module.
Default posture: show less, not more.

Central config created: apps/web/src/lib/role-ui.ts
- Single source of truth for all role-UI rules
- Defines RoleName and NavHref union types
- NAV_BY_ROLE: ordered nav item list per role
- Visibility Sets for commercial fields (proposal CAPEX, margin, opportunity value, pipeline figures)
- All exported as pure boolean functions: canSeeProposalCapex(), canSeeProposalMargin(), etc.

Hook added: useRoleName() in apps/web/src/hooks/use-current-user.ts
- Returns current user's role name string
- Used as the single input to all role-ui functions
