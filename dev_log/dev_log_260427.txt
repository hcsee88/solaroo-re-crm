Dev Log — 2026-04-27
Session: Sales Pipeline Discipline V1 (Activities + Next Action + Health + /sales-pipeline)
=============================================================

CONTEXT
-------
Per spec: improve sales-pipeline visibility WITHOUT integrating email,
WhatsApp, calendar, telephony, or AI. All sales activities remain manual
log entries. Goals: management can answer "which deals are stale", "who
is following up", "what's overdue", "what's closing this month".

This was an extension session, not a redesign. Existing Opportunity model
already had `nextAction` (text) + `nextActionDueDate`. Existing Activity
model already had core fields. Existing SavedViewsBar component was
already built generically. Existing notification + audit + dashboard
patterns were reused.

WORKER ADDITIONS (services/worker/src/index.ts)
=============================================================

Same setInterval pattern as existing daily-digest job — no Redis added.

  SALES_TICK_MS = 60 * 60 * 1000  (hourly)
  RECENT_DEDUPE_HOURS = 22         (re-fire roughly daily)

  notifyOnce() — generic helper that checks for an existing notification
  with matching {userId, type, resource, resourceId} within the last 22h
  and skips if found. Avoids spam without adding a new schema field.

  3 new tick functions:
    tickOverdueNextActions()  — opps with PENDING + due<now → notify
                                opp owner + nextActionOwner
    tickProposalNoFollowup()  — opps in BUDGETARY_PROPOSAL or
                                FIRM_PROPOSAL with no activity in 3 days
                                → notify owner
    tickStaleOpportunities()  — non-terminal opps with no activity in
                                14 days → notify owner

  Notification types: next_action_overdue, proposal_no_followup,
                      opportunity_stale
  Accent colours added in apps/web/src/lib/notification-utils.ts

  All three are no-op when there are zero matching opps.

ASSUMPTIONS / SCOPE DECISIONS
=============================================================

1. OVERDUE next-action-status is computed at read time, not stored.
   Stored values are only PENDING / COMPLETED. The API derives OVERDUE
   from (PENDING && nextActionDueDate < now). This avoids needing a
   worker reconciliation pass, and the UI is always live.

2. Activity ownerUserId is REQUIRED (not optional). Set automatically
   from the calling user — there's no "log on behalf of" feature in V1.

3. Activity delete is restricted to the owner AND only within the first
   7 days of creation. Older activities can only be deleted by DIRECTOR
   or SUPER_ADMIN. Audit-trail integrity > convenience.

4. "High value" filter uses a hardcoded ≥ RM 1,000,000 threshold. If
   business wants this configurable, move to a system setting later.

5. Worker dedupe window is 22 hours, not 24. Tick runs hourly. Means a
   notification fires once per ~24h cycle even if the underlying
   condition persists. Rough but cheap.

6. Worker is NOT deployed to Railway in this commit — only to local
   `pnpm dev`. Production digest + sales ticks won't fire until the
   worker service is properly deployed (separate ticket).

7. Did NOT touch Saved Views internals — reused the existing
   <SavedViewsBar> component as-is. Verified it works generically by
   passing module="opportunities".

8. Did NOT remove the old inline NextActionPanel definition from the
   opp detail page. It's dead code (not referenced) — should be deleted
   in a follow-up cleanup commit. Left it in to keep this diff focused.

9. The OpportunityListItem type in @solaroo/types now requires
   `health`, `lastActivityAt`, `effectiveNextActionStatus`, etc on
   every list response. The API populates these for every row. If any
   other consumer is reading this type from a non-API source, it'll
   need a code update.

10. CONTRACTING stage is included in OPPORTUNITY_STAGE_ORDER but not
    in PROPOSAL_STAGES (only BUDGETARY_PROPOSAL + FIRM_PROPOSAL trigger
    the proposal-no-followup notification). Intentional — once you're
    in CONTRACTING the deal owner is in heavy negotiation and a
    "follow up" nudge is condescending.
